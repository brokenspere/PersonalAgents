High-Level Architecture: Scalable Agent Platform

The platform is designed around a decoupled, Event-Driven Microservices Architecture. This ensures that scaling or adding new workers requires zero modifications to existing components.

System Flow

  1. Scheduler fires a market.open or market.close event based on NYSE market hours.
  2. Message Broker (Redis) receives the event and routes it to the subscribed queues.
  3. ScraperWorker receives the trigger, fetches data from Yahoo Finance, and publishes an Extraction Payload via a news.ready event back to the broker.
  4. NotificationWorker listens for news.ready events, formats the data for specific platforms (Telegram, Discord), and pushes the payload to external APIs.

Core Components

1. Scheduler (Celery Beat)

A standalone container responsible purely for time-based triggers. It emits Events (JSON payloads containing at minimum an ISO-8601 timestamp and context identifiers like {"market": "NYSE"}).

2. Message Broker (Redis)

The central nervous system of the platform. All communication between components goes through Redis, ensuring workers are completely isolated from one another.

3. Workers

Independent components that consume from or publish to the message broker. Each worker is containerized independently.

  • ScraperWorker: Uses BeautifulSoup to extract headlines.
  • NotificationWorker: Handles API communication with Telegram and Discord.
4. Shared Codebase

Common utilities like Celery configuration and data models are shared across independent Worker containers by utilizing Docker build contexts (copying a shared/ directory from the monorepo root), rather than relying on private PyPI packages.
