AWS Deployment Plan (Terraform)

To align with the requirement for independent resource allocation and lightweight containers, the platform will be deployed to AWS using Elastic Container Service (ECS) with AWS Fargate. Infrastructure as Code (IaC) will be managed via Terraform.

Target Architecture

Compute

Amazon ECS (Fargate). Serverless containers allow us to scale the ScraperWorker, NotificationWorker, and Scheduler independently without managing EC2 instances.

Message Broker

Amazon ElastiCache (Redis). A managed Redis cluster to handle Celery queues and message routing.

Container Registry

Amazon ECR. Private registry to store Docker images for our components.

Networking

Amazon VPC. Private subnets for the Redis cluster and Workers, with NAT Gateways allowing Workers to access the internet (Yahoo Finance, Discord, Telegram).

Secrets Management

AWS Secrets Manager. Securely stores API keys and Discord/Telegram webhooks.

Terraform Plan Structure

terraform/
├── vpc.tf          # VPC, Subnets, Internet/NAT Gateways
├── ecr.tf          # ECR repositories for scheduler, scraper, notification
├── elasticache.tf  # Redis cluster configuration
├── ecs_cluster.tf  # ECS Cluster definition
├── iam.tf          # Task execution roles and Secrets Manager access
└── ecs_services.tf # Fargate Task Definitions and Services for each worker

Deployment Strategy

  1. Build & Push: CI/CD pipeline (e.g., GitHub Actions) builds the Docker images from the monorepo root (utilizing the Docker build context for shared/) and pushes them to ECR.
  2. Terraform Apply: Terraform updates the ECS Task Definitions with the new image tags and rolls out the deployment seamlessly.
  3. Resource Allocation: Using Terraform, we can assign specific CPU/Memory limits to individual ECS tasks (e.g., granting the ScraperWorker 0.5 vCPU, while the Scheduler gets 0.25 vCPU).
