Internal Logic & Data Flow

Event Payloads

Every trigger in the system is defined as an Event. A standard event payload looks like this:

{
  "market": "NYSE",
  "timestamp": "2026-06-11T13:30:00Z",
  "event_type": "market.open"
}

This allows workers to understand the exact context of the trigger, even if message processing is delayed.

ScraperWorker Logic (Yahoo Finance)

Triggered by market.open and market.close.

  • Tooling: Uses BeautifulSoup (static HTML parsing) to keep the container footprint lightweight. Playwright is avoided unless static parsing completely fails.
  • Extraction: Scrapes the top headlines and their corresponding URLs.
  • Output: Publishes an Extraction Payload to the news.ready queue. This is raw, unformatted structured data.
{
  "source": "yahoo_finance",
  "market": "NYSE",
  "timestamp": "2026-06-11T13:30:05Z",
  "items": [
    {"title": "Market rallies on open", "url": "https://finance.yahoo.com/news/..."},
    {"title": "Tech stocks surge", "url": "https://finance.yahoo.com/news/..."}
  ]
}

NotificationWorker Logic & Fault Tolerance

Triggered by news.ready.

  • Formatting: Consumes the Extraction Payload and applies platform-specific formatting (e.g., Markdown for Discord, HTML for Telegram).
  • Dispatch: Executes POST requests to external APIs.
  • Fault Tolerance (Delayed Retry): If an external API fails (e.g., rate limits, Discord downtime), the worker does NOT drop the message or send it immediately to a Dead Letter Queue. Instead, it NACKs the event and re-queues it with a delay (exponential backoff) using Celery's built-in retry mechanisms.
